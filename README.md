# docfill

**docfill** — консольная утилита для подстановки значений из одного или нескольких JSON-файлов в шаблоны документов форматов **DOCX**, **ODT** и **DOC**.

Программа ищет в документах плейсхолдеры вида:

`ООО_Ромашка.Руководитель.ФИО.Полные.Родительный`

и заменяет их на соответствующие значения из JSON-файла или из нескольких JSON-файлов.

## Для чего это нужно

docfill полезен, когда нужно:
- автоматически заполнять договоры, приложения, письма, доверенности, карточки организации и другие шаблоны документов;
- хранить реквизиты и типовые данные в одном JSON-файле или раскладывать их по нескольким JSON-файлам;
- избегать ручного копирования одних и тех же сведений из документа в документ;
- выполнять подстановку **локально**, без веб-сервисов и облачных ботов.

## Портируемость

docfill задуман как переносимая консольная утилита.

Если в системе есть **Python**, программа может выполняться там же. Это относится как к **Windows**, так и к **Unix-подобным операционным системам**.

Типичные варианты запуска:
- в **cmd**;
- в **PowerShell**;
- в оболочках Unix-подобных систем, таких как **sh**, **bash**, **zsh** и им подобных.

Иными словами, где есть Python и необходимые зависимости, там docfill и будет работать.

## Поддерживаемые форматы

- **DOCX** — поддерживается напрямую;
- **ODT** — поддерживается напрямую;
- **DOC** — обрабатывается через предварительную конвертацию в **DOCX** с помощью LibreOffice.

## Как это работает

1. Пользователь передаёт программе один или несколько JSON-файлов с данными.
2. Затем передаёт один или несколько файлов документов.
3. docfill находит в документах плейсхолдеры и заменяет их значениями из JSON-файлов.
4. Результат сохраняется в новые файлы рядом с исходными, если не включён режим `--in-place`.

## Формат плейсхолдеров

Плейсхолдер — это заменяемый текст в документе.

Плейсхолдеры представляют собой иерархические ключи, собранные из JSON через точку. Например, если JSON содержит:

```json
{
  "ООО_Ромашка": {
    "ИНН": "1111111111",
    "Руководитель": {
      "ФИО": {
        "Полные": {
          "Именительный": "Иванов Иван Иванович"
        }
      }
    }
  }
}
```

то в документе можно использовать такие плейсхолдеры:

- `ООО_Ромашка.ИНН`
- `ООО_Ромашка.Руководитель.ФИО.Полные.Именительный`

## Несколько JSON-файлов

docfill может принимать на вход несколько JSON-файлов одновременно.

Это удобно, если данные разложены по смысловым частям, например:
- реквизиты организации — в одном файле;
- банковские реквизиты — в другом;
- данные руководителя — в третьем.

JSON-файлы распознаются **по расширению `.json`**. Это обязательное условие для текущей версии программы.

Например, такие имена подходят:
- `company.json`;
- `bank.json`;
- `ООО_Ромашка.docfill.json`.

Если один и тот же плейсхолдер встречается в нескольких JSON-файлах:
- с одинаковым значением — это допустимо;
- с разными значениями — программа завершится с ошибкой.

        "Полные": {

          "Родительный": "Иванова Ивана Ивановича"

## Установка зависимостей

### Python

Нужен Python 3.

### Для DOCX

Дополнительные внешние программы не требуются.

### Для ODT

Дополнительные внешние программы не требуются.

### Для DOC

Для обработки файлов `.doc` нужен **LibreOffice**, потому что старый бинарный формат DOC сначала конвертируется в DOCX.

Если `docfill` не может сам найти исполняемый файл LibreOffice, можно указать его явно через ключ `--libre-office-exec`.

## Запуск

### Windows

```bash
py docfill.py company.json bank.json file1.docx file2.odt file3.doc
```

или:

```bash
python docfill.py company.json bank.json file1.docx file2.odt file3.doc
```

### Unix-подобные системы

```bash
python3 docfill.py company.json bank.json file1.docx file2.odt file3.doc
```

или после выдачи права на исполнение:

```bash
chmod +x docfill.py
./docfill.py company.json bank.json file1.docx file2.odt file3.doc
```

## Справка

```bash
python docfill.py --help
```

## Основные ключи

Поддерживаются один или несколько JSON-файлов, но для распознавания их имена должны оканчиваться на `.json`.


- `-h`, `--help`  
  Показать краткую справку.

- `--ignore-case`  
  Искать плейсхолдеры без учёта регистра.

- `--check`  
  Проверить документы без записи выходных файлов. Полезно для диагностики и просмотра найденных плейсхолдеров.

- `-v`, `--verbose`  
  Показать подробный список замен в формате `было -> стало`.

- `--in-place`  
  Перезаписывать выходные файлы на месте. Для `.doc` это не означает сохранение в исходный бинарный формат: такие файлы всё равно проходят через конвертацию в `.docx`.

- `--suffix SUFFIX`  
  Указать суффикс для выходных файлов. По умолчанию используется суффикс `.rendered`.

- `--libre-office-exec PATH`  
  Указать путь к исполняемому файлу LibreOffice. Этот ключ нужен **только для файлов `.doc`** и **только если программе не удалось самостоятельно найти запускающий файл LibreOffice**, необходимый для конвертации из `.doc` в `.docx`.

## Примеры использования

### Обычная обработка

```bash
python docfill.py ООО_Ромашка.docfill.json договор.docx письмо.odt
```

### Обработка с несколькими JSON-файлами

```bash
python docfill.py company.json bank.json director.json договор.docx
```

### Проверка без записи файлов

```bash
python docfill.py --check ООО_Ромашка.docfill.json договор.docx
```

### Подробный вывод

```bash
python docfill.py -v ООО_Ромашка.docfill.json договор.docx
```

### Поиск без учёта регистра

```bash
python docfill.py --ignore-case ООО_Ромашка.docfill.json договор.docx
```

### Явное указание LibreOffice для DOC

```bash
python docfill.py --libre-office-exec "C:\Program Files\LibreOffice\program\soffice.exe" ООО_Ромашка.docfill.json старый_документ.doc
```

## Выходные файлы

По умолчанию исходные файлы не перезаписываются.

Например:

- `contract.docx` → `contract.rendered.docx`
- `letter.odt` → `letter.rendered.odt`
- `legacy.doc` → `legacy.rendered.docx`

Для изменения суффикса используйте `--suffix`.

## Вывод программы

По умолчанию при успешной обработке docfill показывает количество заменённых плейсхолдеров.

Пример:

```text
[OK] договор.docx -> договор.rendered.docx (заменено плейсхолдеров: 7)
```

При использовании `-v` или `--verbose` программа дополнительно выводит список замен:

```text
[OK] договор.docx -> договор.rendered.docx (заменено плейсхолдеров: 7)
  Список замен:
    ООО_Ромашка.ИНН -> 1111111111
    ООО_Ромашка.Адрес.Юридический -> 123456, г. Москва, ул. Никакая, д. 1, корп. 2, офис 3
```

## Особенности и ограничения

- docfill умеет заменять плейсхолдеры, даже если они разбиты на несколько текстовых фрагментов внутри одного абзаца.
- Если плейсхолдер разорван сложной структурой документа, полем Word, границей отдельных блоков или иным нестандартным объектом, замена может не сработать.
- При замене форматирование обычно сохраняется по первому фрагменту исходного плейсхолдера.
- Для `.doc` требуется LibreOffice.
- В режиме `--ignore-case` нельзя использовать JSON, в котором существуют разные ключи, совпадающие после приведения регистра. Например, `Ромашка` и `РОМАШКА` будут конфликтовать.

## Почему просто не использовать нейросеть?

Потому что не обязательно забивать гвозди микроскопом. К тому же docfill выполняется локально, что является преимуществом перед сетевыми чат-ботами: не нужно переживать за конфиденциальность обрабатываемых данных.

## Где это особенно удобно

docfill хорошо подходит для:
- договоров и приложений к ним;
- доверенностей;
- первичных учётных документов;
- протоколов собраний;
- письменных сообщений;
- любых других шаблонов, для которых многократно применяется одна и та же логика подстановки реквизитов.

## Идея проекта

Проект делает одну конкретную вещь:
**берёт данные из JSON и подставляет их в rich text-документы по плейсхолдерам**.

Без облака, без лишней магии, без ручного копирования реквизитов по десяти шаблонам подряд.
